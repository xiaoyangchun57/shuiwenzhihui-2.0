# 水利智慧运营平台 — 移动巡检方案

> 基于水文大队运维人员实地访谈整理
> 2026-06-26

---

## 一、业务背景

### 1.1 当前模式

水文大队采用**站点周期包干制**：
- 后台管理员按月/半月排程
- 多个运维人员分组，在周期内跑完所有指定站点
- 每站覆盖全部检查项（含设备校验、校准试验）
- 人员自行规划路线

### 1.2 当前痛点

| 痛点 | 描述 |
|------|------|
| 压力前松后紧 | 月底赶工，月初闲着，人天利用率不均 |
| 检查项频次冲突 | 周级项（水位校核）被拉到月，季级项（翻斗校准）每月都做浪费工时 |
| 移动端操作繁琐 | 每项检查都要点开、输入、确认，几十次点击/站 |
| 设备校验输入密集 | 手机小键盘输数字效率低、易出错 |
| 信号差时无法提交 | 野外网络不稳定，频繁卡顿 |

## 二、巡检排程方案

### 2.1 核心逻辑：频次分层 × 站点打包

不改"站点为单位、月窗口、一次跑完"的顶层模式，但**检查项内容按频次分层**：

```
高频层（每次必做） ── 水位校核、设备状态确认、传感器外观清洁
中频层（每月做）   ── 电池检查、太阳能板检查、密封性检查
低频层（每季轮换） ── 翻斗校准、水位计精度校验、流速仪校验
年度层（专项）     ── 全面校准试验，单独排"校准专周"
```

### 2.2 站点分组与路线

> 以下数字为示例说明，实际站点数量、人员数量在系统中配置。

演示计算：假设150站 ÷ 3人 = 每人50站/月，拆成4周分批：

```
第1周：A组站 1–13  ←  路由优化，按地理邻近打包
第2周：A组站 14–25
第3周：A组站 26–38
第4周：A组站 39–50
```

路线由系统按站点经纬度自动推荐，运维人员可以微调。站点分组和人员分配在后台配置中可随时编辑调整。

### 2.3 分层检查项配置

> 检查项的频次要求参照招标文件中规定，同时在系统后台提供可编辑的配置入口，管理员可根据实际运行情况调整。

| 层 | 配置说明 | 频次范围（可配置） |
|----|---------|-----------------|
| 高频 | 每次派单自动附带，不可取消 | 每次巡检必做 |
| 中频 | 周期性出现，按日历周期计算 | 建议周/半月/月 |
| 低频 | 轮换制，多周期内完成一组 | 建议季/半年 |
| 年度 | 专项安排，不混入日常巡检 | 每年一次 |

**关键规则**：系统记录每站上次做了哪项低频检查，自动算下次该做哪项。运维人员不需要记忆。

### 2.4 检查项的自适应跳过

> 根据用户反馈：复杂项可跳过，简单项当场做完，巡检周期相应后延。

**具体规则**：

- **高频/中频项**：原则上不跳过。如果确实因为设备问题无法检查（如设备损坏、正在维修），可标记"暂缓"并在备注说明原因。系统自动延后该站下一次的排期。
- **低频项**：当次到站时，如果该低频项操作复杂（如需要拆装设备、等待稳定时间），运维人员可勾选"跳过本次"，系统将其顺延到下一个轮换周期。如果是简单项（如目视检查、外观确认），鼓励当场完成，完成后该低频项的下次到期日自动后延。
- **累计逻辑**：某站如果连续多次"跳过"同一低频项，系统在第3次时升级为"强制"——下次必须完成，不可跳过。

**系统层面**：每个低频项都有"最近一次完成时间"字段，跳过时只记录"已跳过"不重置计时器，完成时才重置。

## 三、移动端设计（H5 / 小程序）

### 3.1 设计总原则

> **正常为零操作，异常才需要交互。**

一个站如果全部正常，运维人员应该能够在 **3 次点击内完成全站检查**。

### 3.2 主界面：站点检查清单

```
┌─────────────────────────────┐
│  今日巡检 · 邓埠站           │
│  水文站 │ 上次: 06-01        │
├─────────────────────────────┤
│  ☐ 全部正常        [一键通过]│ ← 全站一键正常 + 仅拍一张证照
├─────────────────────────────┤
│ 高频层                      │
│  ✓ 水位校核        17.32m   │
│  ✓ 设备状态确认    正常     │
│  ✓ 传感器清洁      正常     │
│  ✓ 数据通讯检查    正常     │
├─────────────────────────────┤
│ 中频层                      │
│  ✓ 电池电压        12.8V    │
│  ✓ 太阳能板        正常     │
├─────────────────────────────┤
│ 低频层（本轮应做）           │
│  ○ 翻斗校准        待做 ◇   │ ← ◇表示可跳过
├─────────────────────────────┤
│ 年度层（暂时无任务）         │
│  [已完成 0/0]               │
├─────────────────────────────┤
│  📷 现场照片: 已拍 1 张      │ ← "一键通过"之后自动要求拍一张
│  📝 备注:                    │
│                             │
│  [ 提 交 ]  [暂 存]         │
└─────────────────────────────┘
```

### 3.3 关键交互

#### 3.3.1 一键通过（全站正常）

**触发条件**：所有检查项均为正常状态。

**操作流程**：
1. 点击"一键通过"按钮
2. 系统弹出拍照提示："请拍摄现场工作照片作为证明"
3. 拍一张站房/设备全景照片（必拍，补审计留痕需求）
4. 点击"确认提交"

**计数**：1 次点击 + 1 次拍照 = **完成全站**。无需逐项操作。

#### 3.3.2 逐项检查（有异常时）

只有异常项才需要展开：

```
  ✗ 水位校核        17.32m      ← 异常项自动展开
    ┌─────────────────────┐
    │ 遥测值: 17.35m       │
    │ 人工值: [17.32]      │ ← 默认=遥测值，不一致时修改
    │ 误差: 0.03m ✓合格    │ ← 自动算
    │ 异常原因: __________ │
    │ 📷 拍照留证   [已拍]  │
    │ [标记正常] [提交异常] │
    └─────────────────────┘
```

数值型校验使用**步进器**替代数字键盘：

```
人工值: [ − ] 17.32 [ + ]
        步长 0.01m
```

- 默认值 = 遥测值，一致时完全不需要操作
- 不一致时：点 ± 步进调整，或直接点击数值输入
- 建议设置常用值快捷按钮（如"偏低 0.05"、"偏高一档"）

#### 3.3.3 拍照流程（水印照片 + 分类型上传）

运维人员有明确的分类型拍照要求：全景照、设备整体照、设备详细照。每类照片需要拍多张，照片需叠加水印（站点名、时间、GPS位置）。

**移动端照片框设计**：

```
┌─────────────────────────────┐
│ 📷 现场照片（必拍）          │
│                             │
│ [站房全景照]     [已拍 1/1]  │ ✓
│ ┌─────────────────┐        │
│ │  点击拍摄全景照  │        │
│ └─────────────────┘        │
│                             │
│ [设备整体照]     [已拍 2/3]  │
│ ┌────────┐ ┌────────┐      │
│ │ 全景+1 │ │ 正面   │      │
│ └────────┘ └────────┘      │
│ ┌────────┐                  │
│ │  + 拍  │   ← 空白框       │
│ └────────┘                  │
│                             │
│ [设备详细照]   [已拍 0/2]   │
│ ┌────────┐ ┌────────┐      │
│ │  + 拍  │ │  + 拍  │      │ ← 空白框，显示预期照片类型
│ └────────┘ └────────┘      │
└─────────────────────────────┘
```

**设计要点**：

| 特性 | 说明 |
|------|------|
| 分类型照片框 | 每个照片槽位显示其预期类型（"站房全景照"、"设备正面照"、"铭牌特写"等），点击后拍照/从相册选。水印由专用水印相机拍摄时添加，系统不做二次处理 |
| 最少张数要求 | 每种类型标注"最少X张"，未达标时提交按钮显示"N张未拍"，但不强制阻止提交（考虑野外场景） |
| 批量上传 | 所有照片暂存本地，到网后批量上传 |
| 已拍缩略图 | 已拍的照片在框内显示缩略图，可点击查看大图或重新拍摄 |
| 照片压缩 | 宽边 1920px，质量 85% |

**一键通过时的照片要求**：即使全部正常，也需要至少拍1张全景照 + 1张设备整体照作为工作证明。

**异常项时**：按项拍照，每项至少拍1张部位特写，异常项照片单独关联到该检查项记录。照片框由检查项模板动态生成——水位校核、雨量校准等不同项有不同的照片要求。

#### 3.3.4 低频项跳过

```
  ○ 翻斗校准        待做 [跳过] ← 点击后弹出确认
    ┌─────────────────────────┐
    │ 跳过原因：              │
    │ ○ 设备故障无法操作      │
    │ ○ 时间不足下次再做      │
    │ ○ 该设备已停用          │
    │ ○ 其他: ______________  │
    │                         │
    │  [确认跳过] [取消]       │
    └─────────────────────────┘
```

**跳过的影响**：
- 低频项跳过时，计时器不重置，下次到期日后延
- 同一项连续跳过 3 次 → 强制锁定，不可再跳过
- 系统记录跳过次数和原因，供管理层审核

### 3.4 设备校验专项

校验是输入最密集的环节，且不同设备的校验复杂度差异很大。

#### 3.4.1 三类校验模式

按复杂度分三种交互模式：

| 模式 | 适用设备 | 输入方式 | 例 |
|------|---------|---------|-----|
| **单一值校验** | 水位计、地下水位计 | 1个核心值输入 | 人工水位 vs 遥测水位 |
| **多值校验** | 雨量计、蒸发皿 | 多字段输入 + 计算 | 翻斗计数、时间间隔、累计量 |
| **表单校验** | 水质仪、风速仪 | 多参数记录 | 各参数读数逐项录入 |

#### 3.4.2 单一值校验（水位计）

```
┌─────────────────────────┐
│ 水位计精度校验            │
│                          │
│ 遥测值   人工值   误差    │
│ 17.35m  [17.32]   0.03m │ ← 自动计算
│                ✓ 合格   │
│                          │
│ 上次误差: 0.02m           │
│ 趋势: 稳定（连续3次<0.05）│
│                          │
│ [正常]  [异常并拍照]      │
└─────────────────────────┘
```

#### 3.4.3 多值校验（雨量计翻斗校准）

雨量校准不能只填一个值，需要完整的校准流程记录：

```
┌─────────────────────────────┐
│ 翻斗雨量计校准               │
│ 标准: SL/T 246-2025         │
├─────────────────────────────┤
│ 注水量: [10] mm              │ ← 人工注入的标准水量
│ 翻斗次数: [98] 次            │ ← 计数器读数
│ 实际计量: = 注水量 ÷ 次数     │ ← 自动算
│           = 0.102 mm/斗     │
│ 理论值: 0.100 mm/斗         │
│ 误差: +2.0%  ✓ 合格         │ ← 自动算（误差 < 3% 合格）
│                            │
│ ┌─ 校准前 ─┬─ 校准后 ─┐    │
│ │ 0.098   │ 0.101    │    │ ← 历史校准值对比
│ └─────────┴──────────┘    │
│                            │
│ 调整操作:                   │
│ ○ 无需调整                  │
│ ○ 已调整定位螺丝            │
│ ○ 已更换翻斗组件            │
│ ○ 其他: ______________     │
│                            │
│ [正常]  [异常并拍照]        │
└─────────────────────────────┘
```

**雨量校准的特点**：
- 需要注入标准水量 → 记录翻斗动作次数 → 计算单斗计量 → 对比理论值
- 核心不是"填一个数"，而是**记录操作过程和结果**
- 系统应该自动计算误差，给出合格/不合格判断
- 如果操作复杂（需要爬上雨量计平台、拆装防虫网），运维人员可跳过本次

#### 3.4.4 表单校验（蒸发皿、风速仪等）

```
┌─────────────────────────────┐
│ 蒸发皿观测校验               │
├─────────────────────────────┤
│ 蒸发皿类型: E601B            │
│                            │
│ 蒸发前水量: [20.0] mm       │
│ 蒸发后水量: [13.5] mm       │ ← 核心差值
│ 蒸发量    : = 6.5 mm        │ ← 自动算
│                            │
│ 辅助气象要素:                │
│ 气温: [28.5] °C             │
│ 湿度: [65] %                │
│ 风速: [2.5] m/s             │
│                            │
│ [正常]  [异常并拍照]        │
└─────────────────────────────┘
```

#### 3.4.4 校验模板可配置

不同类型的设备校验，其**输入字段、计算公式、合格阈值**都应该做成可配置的：

```json
// 后端配置示例：翻斗雨量计校准模板
{
  "device_type": "rainfall_gauge",
  "fields": [
    {"key": "inject_volume", "label": "注水量", "unit": "mm", "type": "number", "default": 10},
    {"key": "tip_count", "label": "翻斗次数", "unit": "次", "type": "number"},
    {"key": "adjust_action", "label": "调整操作", "type": "select", 
     "options": ["无需调整","已调整定位螺丝","已更换翻斗组件"]}
  ],
  "calculations": [
    {"key": "actual_per_tip", "expr": "inject_volume / tip_count", "label": "实际单斗计量"},
    {"key": "error_pct", "expr": "(actual_per_tip - 0.100) / 0.100 * 100", "label": "误差百分比"}
  ],
  "thresholds": [
    {"key": "error_pct", "max": 3, "label": "误差 < 3% 合格"}
  ]
}
```

这样新增设备类型时只需要在后台配置模板，不需要改代码。

| 特性 | 说明 |
|------|------|
| 默认值 | 数值型字段自动填入遥测值或标准值，一致则无需操作 |
| 步进器 | ± 按钮步进替代数字键盘（步长可配置） |
| 自动计算 | 表达式引擎自动算差值/误差/合格判定 |
| 历史趋势 | 显示该设备历次校验值，辅助判断是否存在漂移 |
| 复杂度分级 | 前端根据配置的字段数量自动选择"简单/标准/完整"三种视图 | 

### 3.5 离线缓存方案

采用 **"Service Worker + IndexedDB"** 离线优先架构：

```
[运维人员操作] → [IndexedDB 本地存储] → [网络可用时] → [批量提交到后端]
                                                    → [冲突检测]
                                                    → [同步结果回写]
```

**具体流程**：
1. 出发前在有网环境下加载"今日任务"到本地
2. 巡检过程中所有操作（勾选、拍照、输入）先存 IndexedDB
3. 拍照暂存本地文件系统（压缩后）
4. 回到有网区域，自动/手动触发批量提交
5. 提交前做增量冲突检测（如模板被后台修改，提示刷新）
6. 成功后清除本地缓存，释放存储空间

**存储估算**：

| 数据类型 | 单站大小 | 150站 | 备注 |
|---------|---------|-------|------|
| 检查记录 | ~2KB | ~300KB | JSON文本 |
| 照片 | ~200KB/张 | ~30MB | 压缩后，按每站1张算 |
| 合计 | ~202KB/站 | ~30.3MB | 手机完全可以接受 |

### 3.6 后端接口设计

```
GET    /inspections/today?user_id=X       → 获取今日巡检任务清单
GET    /inspections/{plan_id}/tasks       → 获取某计划的检查项列表
POST   /inspections/tasks/{task_id}       → 提交单项检查结果
POST   /inspections/{plan_id}/batch       → 批量提交全站结果（含照片）
POST   /inspections/{plan_id}/complete    → 标记全站完成（一键通过）
GET    /inspections/skip-history?site=X   → 获取某站的跳过记录
POST   /uploads/photo                     → 上传照片（压缩+缩略图）
```

所有写接口均支持 `?offline_mode=true` 参数，允许服务端做时序合并。

## 四、派单与期限设定（无专职调派员模式）

### 4.1 当前的矛盾

现在水文大队的做法是：
> 后台管理员主动问运维人员："这次150个站，你们大概需要多少时间？"
> 运维人员回答："大概20天。"
> 管理员就设20天。

这个模式能跑通是因为**有专人负责沟通协调**。智慧运维平台如果只是把"人工问"变成了"系统分配"，但没有专人来根据实际情况灵活调整，那期限就会僵化——要么太紧完不成，要么太松摸鱼。

### 4.2 解决方案：历史工时积累 + 动态拟合

不依赖一个"调度员"角色，而是让系统**通过学习历史数据自动计算合理期限**。

**核心公式**：

```
每站预估耗时 = 该站同类站点历史平均耗时 × 站点距离系数 × 检查项数量系数

总期限 = ∑(每站预估耗时) × 1.15(缓冲系数) + 路程耗时
```

**数据来源**：

| 数据 | 来源 |
|------|------|
| 单站历史耗时 | 往次巡检中"到达时间→完成时间"的实际间隔 |
| 站点距离系数 | 按经纬度计算站点间的路程时间，附加路况因子 |
| 检查项数量 | 该站本次需要做的检查项数量（按频次分层过滤后） |

### 4.3 首次运行（冷启动）

没有历史数据时，系统需要在首次使用时建立基线。系统内置一套按站点类型预设的初始耗时，所有数值在后台可编辑调整。

**初始基线方案（后台可配置）**：

| 站点类型 | 初始预设耗时 | 备注 |
|---------|------------|------|
| 水文站 | 45分钟/站 | 设备多、检查项多 |
| 水位站 | 30分钟/站 | 以水位校核为主 |
| 雨量站 | 25分钟/站 | 含翻斗校准则+15分钟 |
| 蒸发站 | 25分钟/站 | 含气象要素记录 |
| 地下水站 | 20分钟/站 | 下井操作耗时 |
| 墒情站 | 15分钟/站 | 传感器读取 |
| 站院 | 20分钟/站 | 安防+环境检查 |

这些初始值来自行业经验，可以从管理后台配置。

**冷启动时的期限计算**：

```
150站 × 平均30分钟/站 = 75小时
3人分摊 = 25小时/人
每天工作8小时 = 约3.1天/人
缓冲1.15 + 路程0.5天 = 约4天
→ 总期限设为 4天（3人合力）
```

### 4.4 动态迭代：统计修正

每次巡检完成后，系统记录实际耗时，与预估对比：

```
┌───────────────────────────────────┐
│ 工期预估准确度                     │
│                                    │
│ 本月预估: 18天  实际: 20天  偏差+11%│
│ 上月预估: 17天  实际: 16天  偏差-6% │
│                                    │
│ 按运维人员:                         │
│  张建国 预估50h 实际52h 偏差+4%     │
│  黎  明  预估48h 实际55h 偏差+15%   │ ← 标准差偏大，需调整该人员系数
│  王  刚  预估52h 实际49h 偏差-6%    │
│                                    │
│ 按站点:                             │
│  邓埠站 预估45min 实际38min 偏差-16%│ ← 下次调低该站预估
│  泉岭站 预估25min 实际40min 偏差+60%│ ← 校准项太多？调高或检查模板
└───────────────────────────────────┘
```

**自动调整规则**：

| 条件 | 动作 |
|------|------|
| 单站连续3次实际耗时偏离预估 > 20% | 自动更新该站初始预设值 |
| 某运维人员效率显著低于/高于平均 | 生成个人效率系数（如1.2x / 0.8x） |
| 月度总期限偏差 > 20% | 提醒管理员复核检查项配置是否合理 |
| 新站第一次巡检后 | 用实际耗时替换预设值 |

### 4.5 人工干预入口

虽然系统自动算期限，但**管理员仍然可以手动调整**：

- **整体调整**：将本月期限整体±N天（如"雨季到了，统一+5天"）
- **单站调整**：某站因特殊原因调整预估耗时
- **人员调整**：新人入职时手动设置效率系数（默认1.0）
- **暂停/加急**：特殊情况下可临时标记某批次为加急，系统按最短路线重新排程

### 4.6 派单流程汇总

```
管理员下发计划（选择站点 + 人员分组）
        │
        ▼
系统按历史数据计算预估总期限
        │
        ▼
管理员审核期限（可手动调整）
        │
        ▼
系统自动分周排程（按路线优化）
        │
        ▼
运维人员收到任务清单（移动端）
        │
        ▼
巡检执行（耗时被系统自动记录）
        │
        ▼
完成 → 更新历史数据 → 下次排程更准
```

这样既保留了"人工调派"的灵活性（管理员仍然能看到并调整期限），又解决了"没有专人负责协调"的问题——**系统替代了那个专门问"需要多少天"的人**，而且比人问得更准，因为数据是累计的、客观的。
